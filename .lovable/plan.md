

# Smart Quote & Invoice ID Strategy

## The Problem

Today, quote and invoice numbers are generated naively:

```text
Quotes:  Q-0001, Q-0002, Q-0003...    (count + 1)
Invoices: INV-0001, INV-0002...         (count + 1)
Edge fns: QTE-2025-001, INV-2025-001... (different format!)
```

**Issues:**
1. **Import collision** — A client imports 500 invoices numbered `INV-001` to `INV-500`. The next manually created invoice tries `INV-0501` based on count, but if any were deleted or skipped, numbers collide or have gaps.
2. **No separation between imported legacy numbers and Quotr-generated numbers** — they share one column, so the system can't tell which are "ours."
3. **Format inconsistency** — the frontend generates `Q-0001`, the `create-quote` edge function generates `QTE-2025-001`, and imports accept anything. There's no single source of truth.
4. **Race conditions** — using `count + 1` means two simultaneous creates can produce the same number.

## The Strategy: Dual-ID Model

### Concept

Every quote and invoice gets two identifiers:

| Field | Purpose | Example |
|---|---|---|
| `display_number` | What the user sees and controls. Can be their legacy number or auto-generated. Unique per team. | `INV-2024-312`, `Q-Murphy-001`, anything |
| `ref` | System-generated, guaranteed unique, never changes. Used internally for linking, URLs, portals. | `QR-2025-00047` |

**Rules:**
- `ref` is always auto-generated by the backend using a DB sequence — no race conditions, no gaps in the system's own tracking.
- `display_number` defaults to a formatted version of `ref` if the user doesn't provide one, but can be overridden to anything (imported number, custom scheme).
- Existing `quote_number` and `invoice_number` columns become `display_number`. A new `ref` column is added.

### Data Model Changes

```sql
-- Add system ref columns with DB sequences (race-condition proof)
CREATE SEQUENCE quotes_ref_seq;
CREATE SEQUENCE invoices_ref_seq;

ALTER TABLE quotes ADD COLUMN ref TEXT UNIQUE 
  DEFAULT 'QR-' || to_char(now(), 'YYYY') || '-' || lpad(nextval('quotes_ref_seq')::text, 5, '0');

ALTER TABLE invoices ADD COLUMN ref TEXT UNIQUE 
  DEFAULT 'IR-' || to_char(now(), 'YYYY') || '-' || lpad(nextval('invoices_ref_seq')::text, 5, '0');

-- Rename existing number columns for clarity
ALTER TABLE quotes RENAME COLUMN quote_number TO display_number;
ALTER TABLE invoices RENAME COLUMN invoice_number TO display_number;

-- Backfill ref for existing rows
UPDATE quotes SET ref = 'QR-' || to_char(created_at, 'YYYY') || '-' || lpad(nextval('quotes_ref_seq')::text, 5, '0') WHERE ref IS NULL;
UPDATE invoices SET ref = 'IR-' || to_char(created_at, 'YYYY') || '-' || lpad(nextval('invoices_ref_seq')::text, 5, '0') WHERE ref IS NULL;
```

### Frontend Changes

**Quote/Invoice forms:**
- Show "Reference #" field (optional). If left blank, auto-generated on save.
- Tooltip: "Enter your own number or leave blank for auto-numbering"

**Tables and detail sheets:**
- Primary column shows `display_number` (what users care about)
- System `ref` shown as small secondary text only when needed (e.g., portal links)

**Import flow:**
- Imported `invoice_number` / `quote_number` maps to `display_number`
- `ref` is always auto-generated by the DB default — no import can set it

### Backend / Hook Changes

**`useCreateQuote` and `useCreateInvoice`:**
- Stop using `count + 1` for number generation
- Let the DB sequence generate `ref` automatically
- If user provides a `display_number`, use it; otherwise default `display_number` to the generated `ref` value
- Remove the `Omit<..., "quote_number">` constraint — `display_number` becomes optional input

**Edge functions (`create-quote`, `create-invoice`):**
- Same logic: `ref` is DB-generated, `display_number` is optional input or copies from `ref`

**George/agent tools:**
- When referencing quotes/invoices, search by both `display_number` and `ref`

### Validation

- `display_number` must be unique within a team (existing constraint carries over)
- `ref` is globally unique (DB constraint)
- On import, if a `display_number` collision is detected, the existing skip/replace logic applies

1. **Migration SQL** — Create sequences, add `ref` column, rename `quote_number`/`invoice_number` to `display_number`, backfill
2. **`src/hooks/useQuotes.ts`** — Remove count-based generation, accept optional `display_number`
3. **`src/hooks/useInvoices.ts`** — Same
4. **`src/components/quotes/QuoteFormDialog.tsx`** — Add optional "Reference #" input
5. **`src/components/invoices/InvoiceFormDialog.tsx`** — Same
6. **`src/components/quotes/QuotesTable.tsx`** — Update column accessor
7. **`src/components/invoices/InvoicesTable.tsx`** — Update column accessor
8. **`supabase/functions/create-quote/index.ts`** — Use DB default for ref, optional display_number
9. **`supabase/functions/create-invoice/index.ts`** — Same
10. **`supabase/functions/import-data/index.ts`** — Map imported numbers to `display_number`
11. **`supabase/functions/george-webhook/index.ts`** — Search both `display_number` and `ref`
12. **All UI references** to `quote_number` / `invoice_number` — update to `display_number`

## Summary

- **Imported data**: Legacy numbers preserved as `display_number`, system assigns its own `ref`
- **New data**: `ref` auto-generated by DB sequence (no race conditions), `display_number` defaults to same value unless user specifies their own
- **Going forward**: Numbers always make sense — no gaps in system refs, no collisions with imports
